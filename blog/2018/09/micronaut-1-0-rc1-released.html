Micronaut 1.0 RC1 发布，了解其独特的四个关键特性

release 
rain
2018-09-30
评论
本文的作者是Graeme Rocher，他是Grails和Micronaut框架的项目负责人，主导Grails开发10多年，有着丰富的Grails和Spring架构经验。由于Grails和Micronaut的开发团队和社区与Spring项目之间都很有渊源，而我也是其中最早的用户和参与者，所以本文将Micronaut介绍给Spring技术社区，后续也会发布更多相关的技术文章，欢迎大家积极参与讨论和分享。

以下内容来自OCI官方博客，由Graeme Rocher发表：

今天，很高兴OCI的团队宣布推出Micronaut 1.0 RC1。

在一年多的时间里，Micronaut通过支持Java开发人员所熟悉和喜爱的所有功能，例如依赖注入（DI）和面向方面编程（AOP），而并没有在启动时间、性能和内存消耗方面做出妥协，这代表了JVM微服务框架向前迈出的重要一步。

Micronaut 1.0现已完成功能开发（请参阅用户指南，了解RC中的新功能简介），并且比以往任何其他里程碑版本更多，此RC清楚地展示了Micronaut相比传统的Java框架在预先编译（AOT）方面的优势。

反射使JAVA框架更有效率

Java反射API已经存在了很长时间，并且被整个Java生态系统中的库和框架广泛使用。

大多数现代框架使用基于反射的方法来调用端点、创建代理、执行数据绑定、读取配置等。在典型的现代Java应用程序中存在多少动态的、基于反射的逻辑是非常值得注意的，因为语言本身是静态类型的。

不幸的是，使用反射API会带来一些显着的成本，在内存和运行时性能方面，包括：

  • 由于Java中没有通用的反射缓存，并且读取反射数据的成本很高，因此每个库和框架都会生成一个唯一的反射缓存。在典型的现代Java应用程序中，你会发现Spring有一个独特的缓存，Groovy有另一个缓存，另一个是Jackson，等等。这使得优化内存消耗变得极其困难。

  • JIT优化反射调用要困难得多。诸如InvokeDynamic和其他人的努力试图缓解这种情况，但反射性调用永远不会那么有效。

  • 大多数框架都不了解JVM上的各种语言生态系统。Groovy和Kotlin等语言在字节代码级别生成的方法比Java支持其提供的其他语言功能更多。这种差异意味着基于Groovy和Kotlin的类的反射缓存通常更大，因为字节代码包含更多方法、字段等。

我们在开发Micronaut时要实现的关键目标之一是提高Java的效率，使其可行并且可以运行在现代框架功能方面没有任何妥协的应用程序，仅仅占用10兆字节的内存而不是100兆字节。

虽然Micronaut有许多功能无可否认地有利于微服务的创建，但事实上，Micronaut是一个通用框架，可用于构建一系列应用程序，从CLI应用程序到运行在云中的功能，再到任务调度处理来自Kafka的数据的工作。

Micronaut采用定制的DI和AOP模型，不使用反射。

相反，对Java注解处理器API（APT）和Groovy AST的抽象使得开发人员能够构建高效的应用程序，而无需放弃他们熟悉和喜爱的功能。

这种方法打开了无数框架功能的大门，这些功能通过AOT编译更容易实现，并且是Micronaut独有的。

特点1：支持GraavlVM原生镜像

GraalVM是Oracle的一种新型通用虚拟机，它包含一个多语言运行时和一个独特的功能，允许将Java应用程序编译为原生镜像。

由于Micronaut的编译时、无反射的DI和AOP方法，您可以将Micronaut应用程序本地编译为GraalVM原生镜像，允许应用程序以毫秒为单位启动，同时仅占用常规Java应用程序的一小部分内存在JVM上运行。

看看下面这个小动画演示了这个功能：

使用GraalVM原生镜像，Micronaut的启动时间从大约1秒降至大约20ms，而内存消耗从大约60mb（大部分由JVM占用）下降到本机进程的大约20mb。

虽然GraalVM确实有办法激活对传统反射的支持，但这些都是笨拙且次优的。Micronaut为构建与GraalVM兼容的应用程序提供开箱即用的支持，而开发人员无需深入了解哪些类需要反射，哪些不需要反射。

有关此功能的详细信息，请参阅用户指南中有关GraalVM和Micronaut的部分。

特点2：SWAGGER（OPENAPI）的编译时支持

Micronaut 1.0包括对Swagger的编译时支持，Swagger是用于描述API的行业标准工具。大多数集成Swagger的框架都是在运行时使用广泛的基于反射的方法，这些方法在性能和内存消耗方面都存在问题。

Micronaut具有全面的，语言中立的访问者API，允许开发人员执行编译时分析，而不直接依赖于Groovy的AST或Java的APT。Micronaut中的Swagger支持基于此功能。

例如，考虑这个简单的“hello world”示例：

@Controller("/")
@Validated
public class HelloController {
    @Get(uri="/hello/{name}", produces="text/plain")
    public Single<String> hello(@NotBlank String name) {
        return Single.just("Hello " + name + "!");
    }
}

配置完成后，Micronaut将在编译时构建适当的Swagger YAML，可以轻松添加为静态资源，然后可以将其导入到Swagger UI（React/Node应用程序）中，而无需在Java代码中进行任何额外的运行时反射：

paths:
  /hello/{name}:
    get:
      operationId: hello
      parameters:
      - name: name
        in: path
        required: true
        explode: false
        schema:
          type: string
      responses:
        default:
          content:
            text/plain:
              schema:
                type: string

特点3：编译时验证

所有现代Java框架都包含一些您作为开发人员不能违反的继承规则，而像Java这样的语言为您提供了编译时检查机制来确认您的代码是正确的，而大多数现代框架都没有，而是依赖于运行时异常通知用户（他或她）的代码是错误的。

实际上，一个来自静态编译技术使用者的常见批评是，基于注解的编程模型在注解中使用的动态引用是不能做到静态检查的。

尽管这些问题并未阻止基于注解的编程模型成为Java开发人员中最受欢迎的编程模型，但它们可能会导致令人沮丧的体验。这是我们想要在Micronaut的设计中解决的问题。

我们举一个例子，考虑以下简单的路由定义：

@Get("/hello/{name}")
public Single<String> hello(@NotBlank String na) {
     return Single.just("Hello " + na + "!");
}

发现明显的拼写错误吗？

该路由定义了一个名为name的变量，但该方法接受一个名为na的参数。对于大多数现代框架，这将导致运行时错误。

但Micronaut不是：

hello-world-java/src/main/java/example/HelloController.java:34: error: The route declares a uri variable named [name], but no corresponding method argument is present
    public Single<String> hello(@NotBlank String na) {
                          ^

此功能由Micronaut对编译时visitor的广泛支持提供支持，这些visitor可以验证您正确使用框架并为代码中明显的框架相关问题提供早期反馈。

此外，无论您是Java，Kotlin还是Groovy开发人员，此检查机制都可以运行。无论您选择哪种语言，Micronaut跨语言AST的抽象都可确保一致的体验。

特点4：在编译时映射注解（MAPPING ANNOTATIONS AT COMPILE TIME）

我们在Micronaut设计中采用的方法需要创建一组新的Java注解，以便触发AOT编译。

与运行时对框架级注解进行分析的传统Java框架不同，如Java EE（现为Jakarta EE）和Spring，Micronaut执行相同的分析，但是在编译时，允许启动性能更快（特别是随着应用程序的增长）并且内存消耗要低得多。

但是，Java领域中有许多优秀的规范和标准，开发人员已经习以为常; 在Micronaut中支持这些是有意义的。由于Micronaut提供的广泛注解元数据，您可以使用AnnotationMapper在编译时将任何注解的值映射到任何其他注解的值。

这在实践中意味着Micronaut可以支持注解，例如javax.annotation.security.RolesAllowed用于安全性，或者javax.persistence.PersistenceContext用于注入JPA EntityManager。

这个API很简单，以下示例取自RolesAllowedAnnotationMapper：

public List<AnnotationValue<?>> map(AnnotationValue<RolesAllowed> annotation) {
    String[] values = annotation.get("value", String[].class).orElse(new String[0]);

    List<AnnotationValue<?>> annotationValues = new ArrayList<>(1);
    annotationValues.add(
            AnnotationValue.builder(Secured.class)
                            .values(values)
                            .build()
    );
    return annotationValues;
}

如您所见，第1行接收要映射的注解，并且能够返回一个或多个新注解，这些注解映射到内部框架功能（在本例中为安全性）。

我们期望使用此功能在将来支持更多标准Java注解。

总结

我们对Micronaut 1.0非常兴奋，并且相信它处于Java领域中日益增长的AOT编译运动的最前沿。

AOT编译是Java框架的下一个重大创新，因为它允许Java应用程序针对诸如低内存占用微服务，IoT（是在Raspberry Pi上运行的Micronaut），无服务器等场景进行最佳构建。

无论您是构建客户端还是服务器，调度工作或流处理应用程序，Micronaut都能提供一系列引人注目的功能，您无需支付与当前大多数Java框架相关的高额成本。

1.0 RC1的发布只是一个开始，我们在路线图上有更多的内容，包括与Grails更好的集成。我们从热情的Micronaut用户社区得到的响应非常热烈，我们非常期待在接下来的几周内接近最终的GA版本时听取更多反馈意见。
