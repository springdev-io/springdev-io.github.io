Micronaut 1.0 RC2 发布，详细介绍了其实现技术及原理

本文的作者是Graeme Rocher，他是Grails和Micronaut框架的项目负责人，主导Grails开发10多年，有着丰富的Grails和Spring架构经验。由于Grails和Micronaut的开发团队和社区与Spring项目之间都很有渊源，而我也是其中最早的用户和参与者，所以本文将Micronaut介绍给Spring技术社区，后续也会发布更多相关的技术文章，欢迎大家积极参与讨论和分享。

以下内容来自OCI官方博客，由Graeme Rocher发表：

继上周的Micronaut 1.0.0 RC1发布之后，我们很高兴地宣布推出Micronaut 1.0.0 RC2，其中包括社区反馈后的改进（感谢所有报告问题的人！）。

第二个RC提供了另一个机会来测试Micronaut并提供反馈，因为我们建立了GA版本计划，将于10月23日发布 - 正好赶上旧金山的Oracle Code One 2018，我将在大会上展示Micronaut！

此外，如果您在欧洲，旧金山不远，请考虑参加Voxxed Days Paris，我将于10月30日在那里展示Micronaut。

在我之前的文章中，我详细阐述了我们使用Micronaut进行的一些设计选择，尽可能避免使用反射，以及这对开发人员和应用程序的性能有何益处。

在这篇文章中，我将详细阐述传统Java框架面临的挑战以及Micronaut如何解决这些挑战。

Java 优势与不足

Java和JVM提供了一个强大的平台，可以在其上构建可提高开发人员工作效率的框架。诸如注解、反射、创建运行时代理的能力等功能是大多数Java框架工作的主要内容。

不幸的是，开发人员必须处理的一些挑战和限制导致内存消耗或启动时间受到影响，包括：

  • 类型擦除 - Java泛型系统后来在Java的生命周期中添加；由于向后兼容性是必需的，因此结果是类型擦除。处理类型擦除的现有Java框架和工具中的运行时逻辑量是令人兴奋的。

  • 缺少注解元数据 - Java有注解，但Java框架中出现了许多模式，用于使用注解构造型（或元注解），默认情况下，Java API不支持这些注解构造型。因此，计算注解元数据留给每个框架来实现。

  • 无参数名称 - 默认情况下，Java和JVM不保留参数名称信息。可以通过向Java编译器添加-parameters标志来解决此问题，但默认情况下会禁用它。

为什么这些问题对像Spring和Jakarta EE这样的Java框架来说是一个挑战？让我们举一个简单的例子来定义一个接口：

interface HelloOperations<T> {
    @Get("/hello/{name}")
    T hello(@NotBlank T name);

    @Get("/hello-many/{names}")
    T helloMany(@NotEmpty List<T> names);
}

@Controller("/")
class HelloController implements HelloOperations<String> {

    @Override
    String hello(String name)  {
        // logic here
    }

    @Override
    String helloMany(List<String> names)  {
        // logic here
    }
}

HelloOperations接口上的第2行和第3行使用注解定义路径，此外，在name参数上定义@NotBlank约束。（请注意，我使用过Micronaut注解，但同样的例子可以用Jakarta EE或Spring编写）。

HelloController实现接口并提供逻辑。现在，您会认为这个看似简单的示例对于框架开发人员来说很容易处理，但涉及的任务包括：

  • 计算每个方法和控制器类的注解元数据和构造型

  • 遍历HelloController类的类和接口层次结构，以找出name参数上的继承注解

  • 处理泛型和类型擦除要求，该参数在返回类型和参数的类上引入

  • 生成运行时代理以验证@NotBlank约束，该约束反射性地调用HelloController

所有这些只是一个微不足道的例子。随着您添加更多方法，更深层次的继承层次结构，更多接口等，需求变得越来越复杂，并且所有这些需求都必须在运行时处理。

为了支持Java开发人员喜爱的所有这些功能而不会对运行时性能产生负面影响，传统的Java框架会大量缓存，从而导致内存消耗增加，因为这两个问题无法协调 - 您必须在运行时性能低或内存消耗量少之间做出选择。

Micronaut的解决方式

那么Micronaut如何处理这种情况呢？

Micronaut不是在运行时对您的类执行所有这些分析，而是使用预先编译（AOT）在编译时计算所有内容。

通用类型信息

Bean和方法参数的所有泛型类型信息都是提前计算的。例如，要在AOP拦截器内的helloMany方法中检索List的类型参数，您只需执行以下操作：

public Object intercept(MethodInvocationContext context) {
       Map<String, MutableArgumentValue<?>> parameters = context.getParameters();
       MutableArgumentValue<?> namesArgument = parameters.get("names");
       Argument<?> typeArgument = namesArgument.getFirstTypeVariable().orElse(Argument.OBJECT_ARGUMENT);
      ....
}

请注意，参数名称数据存在且未被删除。源代码级别的处理允许Micronaut保留参数名称数据。

此外，Micronaut还将计算类型的类型参数并将它们存储在BeanDefinition中，因此如果您需要计算类型的类型参数，则无需跳过反射箍（reflective hoops）：

BeanDefinition<HelloController> helloDefinition =
                  beanContext.getBeanDefinition(HelloController.class);

List<Argument<?>> typeArguments =
                  helloDefinition.getTypeArguments(HelloOperations.class);

// do something with the type arguments

上面的示例先从BeanDefinition检索出HelloController bean，然后检索用于HelloOperations接口的类型参数，所有这些都不需要昂贵的反射处理。

注解元数据（ANNOTATION METADATA）

常规Java API使您可以跳过箍来检索前一个示例中的注解元数据。您必须遍历类和接口层次结构，反射性地加载每个方法并可能处理可见性问题以检索包含在层次结构中的所有java.lang.reflect.Method实例。

然后，您必须通过查看返回多维数组的getParameterAnnotations()方法来处理每个方法以将所有潜在注解合并在一起，其中每个参数都按照它们在方法中出现的顺序索引。

Micronaut不需要所有这些复杂性，因为注解元数据已在编译时计算：

Map<String, MutableArgumentValue<?>> parameters = context.getParameters();
MutableArgumentValue<?> namesArgument = parameters.get("names");
if (namesArgument.getAnnotationMetadata().hasStereotype(NotEmpty.class)) {
    if (CollectionUtils.isEmpty((Collection) namesArgument.getValue())) {
        throw new IllegalArgumentException("Named parameter cannot be blank");
    }
}

无反射的代理

最后，Micronaut还将在编译时生成一个常规类的代理，而不是通过Java的本机代理机制创建的代理，从而消除了在代理级别进行反射的需要。

这有许多好处，包括：

  • 减少运行时的工作量，改善启动并减少内存消耗

  • 更短，更容易理解堆栈跟踪 - 由于代理直接调用您的代码，因此巨大的框架级堆栈跟踪不是问题

  • JIT更容易优化 - Java JIT比反射调用更容易优化直接调用

  • 无需缓存反射数据 - 读取反射数据非常昂贵，因此大多数框架都会缓存方法引用，从而进一步增加内存需求

  • 更容易与GraalVM兼容 - 尽管GraalVM原生镜像可以使用代理，但这些代理必须提前配置

这可能听起来很复杂，但它为开发人员带来的简单性是一个巨大的胜利。例如，如果您希望实施AOP通知，例如介绍通知，则只需要几个步骤。

例如，假设您希望在编译时从接口实现逻辑。例如，测试框架通常具有用于创建从接口返回替代值的存根或模拟的工具。让我们看看如何在Mironaut中实现存根。第1步是创建注解，例如：

@Introduction
@Type(StubIntroduction.class)
@Bean
@Documented
@Retention(RUNTIME)
@Target({ElementType.TYPE, ElementType.ANNOTATION_TYPE, ElementType.METHOD})
public @interface Stub {
    String value() default "";
}

在第1行，通知被定义为介绍通知，并在第2行给出了一种StubIntroduction。StubInroduction类型应该实现MethodInterceptor接口。以下是一个简单的实现：

@Singleton
public class StubIntroduction implements MethodInterceptor<Object,Object> {

    @Override
    public Object intercept(MethodInvocationContext<Object, Object> context) {
        return context.getValue(
                Stub.class,
                context.getReturnType().getType()
        ).orElse(null);
    }
}

在第6行，实现尝试将给定的值转换为@Stub注解，并将其作为方法调用的结果返回，否则返回null。现在您可以在任何界面上使用@Stub注解：

@Stub
public interface StubExample {

    @Stub("10")
    int getNumber();

    @Stub("Fred")
    String getName();
}

getNumber()方法将返回10，getName()方法将返回“Fred”。就这么简单。没有必要依赖容器来添加这个功能，或者构建一个ProxyFactoryBean实现来在运行时配置任何东西，它就能够工作而且它可以在不使用任何反射的情况下工作。

总结

除了非常适合微服务之外，Micronaut还是一个通用的应用程序框架，具有巨大的潜力，可以彻底改变现代Java应用程序的效率。

通过AOT编译，Micronaut能够预先计算您的应用程序的要求，并在启动和运行之前完成大量繁重工作。这与前几代Java应用程序框架的工作完全不同，它允许Micronaut可以做传统的Java框架通常做不到的事情。
